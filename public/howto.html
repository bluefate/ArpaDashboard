How to

What is automated, what you configure once, and how to register services.

This instance

Live status from /api/health

Does it update Pi-hole automatically?

Yes — when configured. If PIHOLE_URL and PIHOLE_PASSWORD are set in .env, every create / update / delete through the API also upserts or removes a Pi-hole Local DNS record (dns.hosts) for that hostname.

If those variables are empty, the API still saves the service in the dashboard registry, but DNS is not changed — you would add the Local DNS entry in Pi-hole yourself. The Automation status panel above shows which mode this instance is in.

What about Caddy / HTTPS without ports?

DNS only maps a name to an IP. Ports are not part of DNS. For clean URLs like https://shop.dev.home.arpa (no :3333), the name should resolve to your reverse proxy, and Caddy must proxy to the backend.

  • Automated snippet (optional): set CADDY_SNIPPET_PATH to a file your Caddy container can read, and import that file from your main Caddyfile. ArpaDashboard rewrites the snippet whenever services change.
  • Manual: leave CADDY_SNIPPET_PATH empty and add reverse_proxy blocks yourself (same as a typical home-lab Caddyfile).

One-time setup

  1. Copy .env.example → .env.
  2. Generate an API key: 
    openssl rand -base64 32
    Put it in API_KEY. Never commit .env.
  3. Set CADDY_IP to the LAN address clients should resolve for proxied names.
  4. To automate Pi-hole: set PIHOLE_URL (e.g. http://pihole.home.arpa or the Pi’s LAN URL) and PIHOLE_PASSWORD (Pi-hole v6 app password / login password used by the API).
  5. Optional: set CADDY_SNIPPET_PATH and import that file from Caddy.
  6. Start the app (npm run dev or Docker Compose).

Zones and categories

  • dev.home.arpa — laptop / WIP (register form on that page)

Cards are grouped by category (the group field). Use Manage to move a service between categories (including Paused) without re-registering.

Agents / automation: follow the step checklist in /AGENTS.md (health → POST /api/services with proxy: true → verify → ensure reverse proxy).

Interactive API docs

Open /api/docs for Swagger UI. The OpenAPI document is at /api/openapi.json. Use Authorize with your API_KEY to try create / update / delete from the browser.

Register a service

export API_KEY='your-generated-key'

curl -sS -X POST http://127.0.0.1:8787/api/services \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "shop",
    "zone": "dev.home.arpa",
    "ip": "10.0.0.50",
    "port": 3333,
    "title": "Shop admin",
    "group": "Development",
    "proxy": true
  }'

With proxy: true, DNS points at CADDY_IP and (if enabled) a Caddy route targets ip:port. With proxy: false, DNS points at ip directly (useful for SSH or non-HTTP).

The JSON response includes an integrations object so you can see whether Pi-hole and Caddy steps ran, were skipped, or failed.

Enable Pi-hole DNS automation

Set these in your local .env (or the env file your Compose stack mounts). Never commit secrets. 

PIHOLE_URL=http://192.168.1.2
PIHOLE_PASSWORD=your-pihole-password

Then restart the app:

# Docker Compose
docker compose up -d --force-recreate

# or local Node
npm run dev

After restart, /api/health (and the dashboard status panel) should show Pi-hole as automated. Use the Pi-hole web UI password (or an app password if you use one).

Uninstall / remove

ArpaDashboard does not ship an uninstall binary. Removal is stopping the app and optionally cleaning the records it created. Pick how far you want to go.

1. Stop the app only (keep data)

Local Node:

# stop the terminal process (Ctrl+C), or if you used a process manager, stop that service

Docker Compose:

docker compose down
# container removed; ./data and .env remain on disk

2. Remove local files

# from the project directory
rm -f .env
rm -rf data/services.json
# optional: remove the whole checkout
# cd .. && rm -rf ArpaDashboard

Deleting the app does not automatically delete Pi-hole Local DNS entries or Caddy routes. Those live outside this project.

3. Clean up services while the API still runs (recommended)

Before you tear the app down, delete each registered service so Pi-hole (and the Caddy snippet, if enabled) are updated: 

export API_KEY='your-key'
# list ids
curl -sS http://127.0.0.1:8787/api/services
# delete one
curl -sS -X DELETE http://127.0.0.1:8787/api/services/SERVICE_ID \
  -H "Authorization: Bearer $API_KEY"

If Pi-hole was never configured, delete matching Local DNS rows in the Pi-hole UI (Settings → Local DNS) yourself.

4. Reverse proxy / portal

  • Remove any reverse_proxy (or import) you added for ArpaDashboard or generated hostnames.
  • If you pointed home.home.arpa at this app, point it back at your previous static site or another upstream, then reload Caddy.
  • If you used CADDY_SNIPPET_PATH, delete that snippet file and drop its import from the Caddyfile.

5. Full wipe checklist

  1. DELETE each service via the API (or remove Local DNS rows manually).
  2. docker compose down or stop the Node process.
  3. Remove project files / compose stack from the NAS or host.
  4. Revert Caddy routes and reload Caddy.
  5. Confirm leftover names with dig hostname @<pihole> and remove stragglers in Pi-hole if needed.

Keep ArpaDashboard on your LAN or VPN. Do not expose the API to the public internet.
